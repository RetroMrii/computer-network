How the server works

From a TCP connection to a complete HTTP/1.0 response

This server is built as a learning model of the web stack. It does not hide the network layer behind a web framework: it opens a socket, accepts client connections, reads raw bytes, interprets the HTTP text, chooses a static file, and writes response bytes back to the client.

1. Socket setup

The server creates a TCP socket, binds it to the selected host and port, listens for clients, and accepts incoming connections.

2. Threaded handling

Each accepted client is handed to a separate worker thread so one slow connection does not stop other clients from being served.

3. Stream buffering

The worker reads from the socket repeatedly until the HTTP header terminator arrives. This matters because TCP is a byte stream, not a message protocol.

4. Manual parsing

The request line is interpreted as text and split into the method, path, and HTTP version without using a routing engine or web-server library.

Request line anatomy

The request line has three meaningful fields. The method decides whether the request can be served, the path decides which static file is requested, and the version identifies the HTTP syntax.

GET /public/index.html HTTP/1.0

Safety and correctness

Every path is checked before any file is opened

The server decodes URL paths, rejects traversal attempts, and only serves files from the approved static roots. This keeps a request from escaping the website directory or reading private local files.

  1. 1 Decode the URL path
  2. 2 Reject .. traversal
  3. 3 Check the first directory
  4. 4 Resolve the path inside cna/
  5. 5 Serve only existing static files

Static files only

The server returns HTML, CSS, JavaScript, and SVG files as static bytes. There are no templates, sessions, cookies, databases, or backend API routes.

Clear response codes

Successful files receive 200. Malformed or unsafe paths receive 400. Blocked directories receive 403. Missing files receive 404. Unsupported methods receive 405.

HTTP response structure

Every response is assembled with a status line, headers, a blank line, and a body. Content-Length tells the browser exactly how many bytes are in the response body.

One request, one response

After sending the response, the server closes the connection. This keeps the design stateless and makes each request independent.
