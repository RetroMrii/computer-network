Socket lifecycle

From TCP connection to HTTP response

The backend is intentionally low level: it accepts a socket, reads bytes until the header is complete, parses text by hand, and returns a single static-file response.

1. Accept

The listening socket accepts a client connection and hands it to a worker thread.

2. Buffer

The worker keeps reading bytes until the HTTP header terminator arrives.

3. Parse

The request line is split into method, path, and version.

4. Serve

Allowed static files are returned with HTTP/1.0 headers and raw body bytes.

Request line anatomy

The parser expects exactly three fields. Non-GET methods are syntactically valid, then rejected later with 405.

GET /public/index.html HTTP/1.0

Manual response bytes

Responses include a status line, Content-Length, Content-Type, Connection: close, a blank line, and body bytes.

Stateless by design

Each connection handles one request. No cookies, sessions, templates, routes, or backend APIs are involved.
